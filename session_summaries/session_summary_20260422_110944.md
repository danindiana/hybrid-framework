# Session Summary - 2026-04-22 11:09:44

## Objective
Investigate network downtime and unauthorized or unexpected changes to `/etc/resolv.conf` (reported as `/etc/rtesolv.conf`).

## Findings

### 1. DNS Resolution Failure (Not Internet Downtime)
- **Status:** The internet connection was technically active (pinging 8.8.8.8 succeeded), but DNS resolution was completely broken from boot (**10:25 AM**) until the PIA VPN connected (**10:50 AM**).
- **Cause:** `systemd-resolved` was in **uplink mode** instead of the standard stub mode.
- **Trigger:** The recently used `wg-quick` tool (for the `BWD` WireGuard interface) and the `resolvconf` package interacted to modify the behavior of `systemd-resolved`.
- **Conflict:** `dnsmasq` started at boot and bound to `127.0.0.1:53`. This prevented `systemd-resolved` from starting its standard **DNSStubListener** (which also requires port 53). As a result, the standard `/etc/resolv.conf` path was diverted to the uplink file.
- **Outage Duration:** Because the uplink file was empty or pointing to unreachable local addresses (like `192.168.3.1` from the `BWD` config) until the VPN established a connection, the system had no working DNS for approximately 25 minutes.

### 2. Multipath Routing Instability
- **Rogue Route:** A multipath default route was detected with metric 0 (highest priority):
  `default nexthop via 192.168.1.254 dev enp3s0f0 weight 1 nexthop via 192.168.1.254 dev enp3s0f1 weight 1`
- **Context:** This exact configuration was identified as a cause of instability in a previous triage session (March 27). It was supposedly removed but has returned, likely via a boot script or NetworkManager misconfiguration.
- **Impact:** Load-balancing across two physical ports caused instability for VPN tunnel packets and stateful connections.

### 3. Resolution
- **VPN Connection:** At **10:50:26 AM**, Private Internet Access (PIA) successfully connected via WireGuard (`wgpia0`).
- **DNS Restoration:** Upon connection, PIA pushed its DNS servers (`1.1.1.1`, `8.8.8.8`) to `systemd-resolved`. This updated `/run/systemd/resolve/resolv.conf`, immediately restoring DNS functionality for the system.
- **Verification:** `ntpd` logs confirm that name resolution started succeeding at exactly **10:50:32 AM**.

## Actions Taken (Permanent Fixes)
1.  **Disabled dnsmasq:** Port 53 was previously occupied by an empty `dnsmasq` instance, blocking the standard system resolver.
2.  **Restored Resolver Stub:** 
    - Removed `/etc/systemd/resolved.conf.d/no-stub.conf` which was forcing "uplink" mode.
    - Standardized the `/etc/resolv.conf` symlink to point back to the local stub (`127.0.0.53`).
3.  **Removed Rogue Route:** Deleted the metric 0 multipath default route that was causing unstable load balancing across the physical interfaces.
4.  **Verification Script:** Created `check_network.sh` to allow the user to verify these three critical points after future reboots.
