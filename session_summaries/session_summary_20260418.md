### Status & Observation
Upon reboot, the system fails to display the expected text-based login prompt on TTY1. Instead, it proceeds to a blank screen. Investigation reveals that the system is actually performing an **automatic login** for the user `jeb` and immediately initiating an X session (`startx` -> `xfce4-session`) on TTY1.

### Technical Findings
1.  **Autologin Detected**: `journalctl` logs show `getty@tty1` starting at `14:36:47` and a session for `jeb` being opened at `14:37:02` without manual intervention.
2.  **X Session Trigger Located**: The file `~/.zprofile` was discovered to contain:
    `[[ -z "$DISPLAY" && "$(tty)" == "/dev/tty1" ]] && exec startx`
    This causes `zsh` to automatically initiate the GUI session immediately upon login on TTY1.
3.  **Timeline Confirmed**: Analysis of `last` logs indicates this behavior began between **April 16 (18:19)** and **April 18 (13:23)**. Previous boots showed a 60+ second delay (manual login), while recent boots show a 15-second automated sequence.
4.  **Display Failure**: The "blank screen" reported by the user is likely X failing to render properly on the current display configuration, potentially due to the NVIDIA BusID/dual-monitor issues identified in previous system repairs.
5.  **Suspicious Activity**: A `sudo lsof -i :3001 -t` command was executed on TTY1 at the exact moment of login, suggesting a script is running at startup that may be responsible for the `startx` call.
